员工班车超低载线路优化算法

发表于 2019-03-14

优化目标

通过优化班车开行线路，保证员工有车可乘的前提下，节约班车运营成本。

  • 不增加班车车次的情况下，尽可能多地保证超载线路/站点的员工乘车需求。
  • 保证员工正常乘车的情况下，尽可能裁撤上座率过低的线路/站点

优化方案

  • 超载站点的员工能够有车可乘
  • 提高普通低载线路（站点）的上座率（$P$）
  • 裁撤整个极低载线路（$RL$）

限制条件

  • 绕行距离不超过 3KM
  • 核载在 70%-80% 以内，即 50 座的车 35-40 人以内
  • 大站发车发车时间应该在 5 分钟内
  • 超过 40 人为大站，可直接快车直达
  • 25 人以上开新线
  • 低载站点：低于 5 人考虑撤销或者合并站点
  • 低载线路：时间粒度 80%以上，低于 20 人的（50 座的车，40%）
  • 800 米内有的站点不设置新站点
  • 上班线时间不能晚于 9:10

假定和特点

  • 车辆到站后，乘客上车立即出发。车辆不在站点处候车。粗略认为到站时间与发站时间一致。
  • 上班班车直至沿途只上客不下客，直至终点；下班班车从起点开始只下客不上客。

输入、输出

输入：

  • 所有线路信息
    • 车辆总容量
    • 途径站点，每个站点的 GPS 位置、上客时间、每天平均上客人数、合计上客人数（去重）

外部依赖：

  • 地图 Api：
    • 根据指定站点获得一定距离范围内的其他站点
    • 指定几个站之间的行车距离

输出：

  • 优化结果：
    • 超载线路：站点 A/B/C… 分别超员人数已被线路 R1/R2/R3… 接走。
    • 普通低载线路：绕行站点 S1、S2、S3… 并接送乘客人数，示例：
      • 优化前途经：A – B – C – D
      • 优化后途经：A – B – S– C – D
      • 在站点 S 处上客时间和人数
      • 当前线路新状态：正常/仍然低载
    • 极低载线路：可裁撤的站点（上客人数变为零的站点），示例：
      • 优化前途经：A – B – C – D
      • 优化后途经：A –(0)–(0)– D
      • 当前线路新状态：部分站点裁撤/整条线路裁撤

算法简述

我们会将所有的班车线路分为两大类三小类（忽略正常线路），对于这三小类线路，我们将采取不同的策略进行优化：

  • 超载线路（$O$）
    • 不改变原本线路及其站点，仅对超载的站点进行局部优化，通过算法查找最适合到该站点接人的低载班次前来接人，从而满足超员部分的乘车需求。
    • 优化后，超载线路不再超载。算法无法优化的站点，作为结果返回。
  • 低载线路（$L$）
    • 普通低载线路（$LN$）
      • 作为机动，绕行到超载站点接送员工的班车线路。
      • 优化后，载荷逐渐增加直至达到最大允许上座率 $PB$。无论线路是否已改版，均作为结果返回。
    • 极低载线路（$LE$）
      • 由于整条线路每个站点的上座率都较低，准备裁撤的线路。
      • 我们认为可以与超载线路等效处理，认定极低载线路各个站点的实际乘车人数即为超员人数即可，直到其他车次接走所有超员人数，该线路即可进行裁撤。
      • 优化后，线路的所有站点或部分站点被裁撤，可以作为结果返回。

我们将循环遍历每条超载线路，针对这两种线路上的站点，查找经过该站点附近满足一定条件的普通低载线路，让这些线路去接送该站点的乘客。超载线路遍历完成之后，再遍历极低载线路，并且在处理过程中，预期裁撤，当做超载处理。

在循环过程中，每次使用普通低载线路去绕着接送超载线路乘客时，都会将相关站点数据更新到原始线路上，进入下一次迭代。例如，普通低载线路 $LN_1$ 绕行了站点 $S_n$，则在下一次循环迭代线路时，将会继承该站点，考虑为 $LN_1$ 线路有途径站点 $S_n$，并且在该站点上客 $X_n$ 人。

最终，超载线路的乘客被普通低载线路接走，极低载线路当做超载线路处理，部分普通低载线路因为绕行接人成为正常线路。

算法细节

计算待优化站点的供需矛盾指标 $X_n$

超载线路上有若干的站点，具体从哪里开始超载，需要优先优化哪个站点呢？

假定所有超载站点都需要进行无差别优化，所有超员乘客都需要接送，对于待优化站点则无需排序，从起始站点开始依次优化即可。但考虑实际上优化幅度可能受限，无法大幅度调整，只进行局部优化，因此我们应当找出矛盾最尖锐的站点来进行优化，因此需要对待优化站点进行优先级排序。

对于线路$R$，令：

  • 满载容量为 $C$
  • 共有站点数为 $e$，
  • 从起点到终点的站点序列为：$S_1$, $S_2$, $S_3$, … , $S_n$, … , $S_e$
  • 各站点未能满足的乘车需求人数为：$X_1$, $X_2$, $X_3$, … , $X_n$, … , $X_e$

很显然，车辆的运力 $C$ 是一定的，而各个站点期待上车的人数可能是不一致的。对于超载的情况，需求大于供给，需求越旺盛，矛盾越激化。我们可以通过 $X_n$ 来作为供需矛盾的量化指标，对该指标进行排序，即可得到站点的重点优化顺序。

下面开始进行 $X_n$ 量化分析的过程。

统计一段时间内（$Period$）的乘车情况，则有：

  • 各站历史上车人数（取 $Period$ 内所有上车人数的并集去重）：$U_1$, $U_2$, $U_3$, … , $U_n$, … , $U_e$
  • 各站平均上车人数（取 $Period$ 内每日实际刷卡上车的平均数）：$A_1$, $A_2$, $A_3$, … , $A_n$, … , $A_e$

从含义上讲，$U_n$ 表示在站点 $S_n$ 曾经乘坐过班车的总人数，其中一部分为常客，一部分为临时或偶尔乘坐班车的乘客；而 $A_n$，则可以认定为经常乘坐并且经常能够坐上车的乘客。很显然有：

$$U_n ≥ A_n$$

另一方面，对于 $A_n$ 的值，令班车出站 $S_n$ 时剩余的空座位数为 $E_n$，则班车入站时剩余的空座位数为 $E_{n-1}$，则有：

$$A_n ≤ E_{n-1}$$

考察 $E_{n-1}$，会有如下几种状态：

  1. $U_n$ < $E_{n-1}$

    这种状态表示车辆入站时剩余座位总能满足乘客需求，此时应当认定为未超载状态，并且有：

    $$E_n = E_{n-1} - A_n$$
    $$X_n = E_n * -1 < 0$$

  2. $A_n$ << $E_{n-1}$

    在这种状态下，站点 $S_n$ 的常坐乘客只有 $A_n$，并且几乎都能满足需求。

    亦即，即使 $C$ < $E_{n-1}$ + $U_n$ 成立，也不能说明该站点已经超载，应当认定为未超载状态。并且有：

    $$E_n = E_{n-1} - A_n$$
    $$X_n = E_n * -1 < 0$$

  3. $A_n$ ≈ $E_{n-1}$

    在这种状态下，该站点平均乘坐人数都接近剩余空座数，则表示在该站点的载荷接近饱和，此时应当认定该站点为超载状态。很显然，班车出站时：

    $$E_n = 0$$
    $$X_n = U_n - A_n $$

    由其他线路来解决超载问题时，应当将 $X_n$ 都接走。

    此时可以设定一个比例 $p\%$，认为只需要接走一定比例的乘客即可满足业务需求，则 $X_n$ = $U_n$ * $p\%$ - $A_n$。

对于起点站有 $E_0$ = $C$，从起点站进行推导，可以得到所有站点的 $X_1$, $X_2$, $X_3$, … , $X_n$, … , $X_e$，现在对 $X_n$ 按大小降序排列，即可得到站点矛盾激化程度，决定站点优化的顺序。

按 $X_n$ 权重顺序优化超载站点

对于超载站点 $S_n$，首先查找附近的站点作为“候选站点”，期待候选站点的车辆过来接送超载的部分员工。根据地理位置，可能找到多个候选站点，此时算法应当给出一个量化标准，衡量哪个站的哪辆车来接送才是最优解。

仅考虑普通低载线路前来接送乘客。对于超低载线路，期待裁撤，按照超载处理。

  • 令当前需要处理的超载站点为：$S_n$
  • 令当前超载车次抵达 $S_n$ 的时刻为：$T_n$
  • 令找到的候选站点数为：$k$
  • 令候选站点的集合为：$S_{n,1}$, $S_{n,2}$, …, $S_{n,k}$
  • 令候选站点车辆抵达对应站点的时刻为：$T_{n,1}$, $T_{n,2}$…$T_{n,k}$

令候选线路量化评分为 $Φ$，考虑 $Φ$ 与如下几个要素有关：

  • A. 起点或目的地评分：分值为 1 或 0。

    对于上班（下班）班车，目的地（起点）应当一致。一致记分值为 1，否则分值为 0。

    考虑多目的地（起点）串联，如上班班车经过科兴再到腾大，或下班班车从朗科经过飞亚达、腾大、科兴之后再出发的情况。待考察的线路经过当前站点的目的地（起点）也可以计分为 1。

  • B. 时间差：得分区间为 [0, 100]

    候选车辆到站点 $S_{n,k}$ 的时刻，与当前站点到站乘车时刻 $T_n$ 的时间差，不得大于 $ΔT_{max}$，此时有：

    $$ΔT_{n,k} = T_n - T_{n,k}$$
    $$ | ΔT_{n,k} | < ΔT_{max}$$
    $$T_n - ΔT_{max} < T_{n,k} < T_n + ΔT_{max}$$

    根据实际情况，当 $ΔT_{n,k}$ > 0 时，即候选站点的到站时间早于当前站点 $S_n$ 的到站时间，为尽可能地不延误乘客上车时间，更适合前来接站，此时评分较高；当 $ΔT_{n,k}$ < 0 时，虽然时间延迟，但仍然在 $ΔT_{max}$ 范围内，此时评分较低。

    根据上述情况，可以根据 $T_{n,k}$ 在数值域 [$T_n - ΔT_{max}$, $T_n - ΔT_{max}$] 的位置情况进行评分，值越小，分越高。

  • C. 时间顺延：得分区间为 [0, 100]

    考虑上述时间差的存在，同时由于候选班车绕行接送当前站点 $S_n$ 的乘客，势必因为往返而多消耗时间。我们需要估算这个时间，用于评估整条候选线路抵达终点时，延时最多不超过 $ΔT_{sum(max)}$。

    候选班车调整前开行站点顺序为 $S_{n,k}$ -> $S_{n,k+1}$，调整后开行站点顺序为 $S_{n,k}$ -> $S_{n}$ -> $S_{n,k+1}$，我们将指定候选线路因为第 $x$ 次绕行导致的时间延迟记为 $ΔT_x$，则有：

    $$ΔT_x = f(S_n, S_{n,k}, S_{n,k+1})$$

    其中，函数 $f$ 可以是通过地图 Api 获取预测时间、通过绕行距离/流程、绕行平均速度计算时间等。

    $$
    f=\begin{cases}
    地图 API 预测从 S_n 经过 S_{n,k} 到 S_{n,k+1} 的耗时 \
    \frac{从 S_n 经过 S_{n,k} 到 S_{n,k+1} 的距离}{平均速度}\end{cases}
    $$

    最后，累计所有 $ΔT_x$，即可得到整条线路在调整后预计的时间顺延：

    $$\sum_{x=1}^{N}|ΔT_x| < ΔT_{sum(max)}$$

    实际上，对于候选站点以后的所有站点，其上车时间均应当往后调整 $\sum_{x=1}^{N}|ΔT_x|$

  • D. 距离：得分区间为 [0, 100]

    为尽可能小地调整，尽可能小地影响现有班车站点及乘车时间，我们应当优选距离当前站点最近的候选车辆前来接送乘客。令 $S_{n,k}$ 与 $S_n$ 之间的距离为 $L_{n,k}$，并且该距离不得大于 $L_{max}$，则有：

    $$0 < L_{n,k} < L_{max}$$

    根据实际情况，距离越近，得分应当越高。

    对于 $L_{n,k}$ 的计算，可以考察两点之间的直线距离、行车距离、当前站到经行路线的最近距离等，有条件可以再结合交通路况进行评估。

  • E. 剩余座位数：得分区间为 [0, 100]

    候选车辆应当有剩余的座位，才有前来接站的价值。并且尽可能地由一辆车将当前站点所有超员的乘客一次性接走为宜。因此我们需要考察候选车辆的剩余座位数 $ω$，考察 $ω$ 和 $X_n$ 的大小关系：

    • 0 < $ω$ < $X_n$

      此时表示剩余座位不足以一次性接走所有乘客，该次车不应当作为优选，并且 $ω$ 越接近 0，评分越低。

    • $X_n$ < $ω$

      此时表示剩余座位可以一次性接走所有乘客，该次车应当作为优选，同时 $ω$ 越接近 $X_n$，评分越高。

    基于以上情况，可以根据剩余座位数为当前候选站点进行评分。

    注：$ω$ 应当取候选车辆抵达终点时的剩余座位数。

  • F. 绕行站点数：得分区间为 [0, 100]

    根据规则，候选车辆不得绕行过多站点。根据候选车辆已经绕行的站点数与最大绕行站点数阈值 $N$ 比较，绕行越少，评分越高。

根据以上要素，对 B~F 进行加权综合评分，得到：

$$Φ = A (B φ_b + C φ_c + D φ_d + E φ_e + F φ_f)$$

对 $Φ$ 进行降序排列，即可得到最优的候选线路。

以上，对所有待优化线路迭代计算之后，即可以得到整个的优化方案。

gRPC Gateway

如果我们的 gRPC 服务希望通过 HTTP 协议公开出来，以供现有的其他服务调用，那么我们就需要用到 gRPC Gateway。它是 Google 官方提供的一个反向代理，核心功能是提供 HTTP 接口，将接收到的 JSON 请求解码再编码为 pb 二进制格式，然后通过 gRPC 调用服务端，服务端返回了 pb 二进制的响应之后，它再次解码编码为 HTTP 响应体返回到客户端。

本文为 QCon 2018 上海站主题演讲嘉宾、Heptio 资深工程师、著名 Go 语言专家 David Cheney 关于 Go 语言实践的英文分享。为方便大家阅读，在此由 Austin Luo 翻译为中文，在文中难以理解之处，也特别增加了译者的理解说明。翻译水平有限，如有偏颇之处，烦请联系我（uonun@163.com）更正。转载请注明出处，保留本节译者注。

目录

  • 指导原则
    • 简单性
    • 可读性
    • 生产率
  • 标识符
    • 选择清晰的名称，而不是简洁的名称
    • 标识符长度
    • 命名中不要包含所属类型的名称
    • 使用一致的命名风格
    • 使用一致的声明风格
    • 成为团队合作者
  • 代码注释
    • 变量和常量上的注释应当描述它的内容，而非目的
    • 总是为公开符号写文档说明
  • 包的设计
    • 一个好的包从它的名称开始
    • 避免将包命名为 base、common、util
    • 快速返回，而不是深层嵌套
    • 让零值变得有意义
    • 避免包级别的状态
  • 项目结构
    • 考虑更少，更大的包
    • 确保 main 包越小越好
  • API 设计
    • 设计难以被误用的 API
    • 针对默认用例设计 API
    • 让函数自身定义它所需的行为
  • 错误处理
    • 通过消除错误来消除错误处理
    • 错误只处理一次
  • 并发
    • 保持自己忙碌，否则自己做
    • 将并发留给调用者
    • 不要启动一个永不停止的协程

Understanding Architecture

发表于 2018-09-30

架构是这样定义的

  • 每个系统都有一个架构
  • 架构由架构元素以及相互之间的关系构成
  • 系统是为了满足 利益相关者（stakeholder） 的需求而构建的
  • 利益相关者都有自己的关注点（concerns）
  • 架构由架构文档描述
  • 架构文档描述了一系列的架构视角
  • 每个视角都解决并且对应到利益相关者的关注点。

架构师的首要任务是尽最大可能找出所有利益相关者，业务方，产品经理，客户/用户，开发经理，工程师，项目经理，测试人员，运维人员，产品运营人员等等都有可能是利益相关者，架构师要充分和利益相关者沟通，深入理解他们的关注点和痛点，并出架构解决这些关注点。架构师常犯错误是漏掉重要的利益相关者，沟通不充分，都会造成架构有欠缺，不能满足利益相关者的需求。利益相关者的关注点是有可能冲突的，比如管理层（可管理性）vs技术方（性能），业务方（多快好省）vs 技术方（可靠稳定），这需要架构师去灵活平衡，如何平衡体现了架构师的水平和价值。
